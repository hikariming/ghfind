**一句话：** 这个分数只回答一个很实际的问题——*这个 GitHub 账号是不是一个真实、有价值的开发者，还是被包装成像开发者的样子？* 而且它每次都用同一套规则、只看公开数据、把所有规则都摆在明面上来回答。这篇文章不讲黑话，把这个数字是怎么算出来的讲清楚。

## 为什么要有一个分数

越来越多的判断，就靠瞄一眼某人的 GitHub。招聘的人在电话前先扫一遍主页；维护者要决定一个陌生人的 Pull Request 值不值得看；目录类网站按"看上去有多牛"给账号排序。每一种用法，都在制造伪造这些信号的动机——而便宜的信号，造起来太容易了。star 能买，粉丝能互换，一个下午开一百个只改一行的 Pull Request，就能自称"开源贡献者"。

所以一个有用的分数，不能只是把那些又大又亮的数字加起来。它必须重重压在真正难造假的东西上，对造假成本低的东西则不太当回事。下面所有的设计,都是这一个想法推出来的。

## 唯一的原则：难造假的，才给重分

把 GitHub 上的每个信号分成两桶。

- **造假便宜的：** star、粉丝。几块钱或者一个互关圈就能刷出来。
- **造假昂贵的：** 合并进**别人**维护的真实项目里的 Pull Request、常年稳定的活跃、被一个很忙的维护者真正接受了的代码。

引擎给第二桶重重加分，给第一桶轻轻加分。star 和粉丝仍然算数——一个真的很受欢迎的项目**理应**帮到你——但它们的上限压得足够低，买来的那点几乎推不动指针。而把真实的代码合并进一个知名仓库，需要说服一个没有任何理由帮你的人，这是整块记分板上分值最高的事。

整套哲学就这么简单。剩下的，只是它怎么摊到六个类别里去。

## 它量的六件事

分数从 0 到 100，分摊在六个维度上。下面是每一个的大白话解释，附上满分。

| 维度 | 满分 | 它其实在问什么 |
|---|---|---|
| **贡献质量** | 27 | 你有没有把真实的 PR 合并进真实的项目，维护者接不接受？ |
| **生态影响力** | 20 | 你的代码有没有落进真正热门、而且不是你自己的仓库里？ |
| **原创项目质量** | 18 | 你有没有做出别人真的在用的东西（用 star 衡量，但有上限）？ |
| **活跃真实性** | 17 | 你是长期、多样地稳定活跃，还是爆发一下就没声了？ |
| **账号成熟度** | 10 | 这个账号存在了多久、活跃了多久？ |
| **社区影响力** | 8 | 你有没有真实的关注者，关注/被关注比健不健康？ |

![100 分都摊到了哪里，按维度](/blog/how-we-score-github-accounts/weight-breakdown.svg "六个维度及其满分。橙色是难造假的信号；灰色是能买来的。")

注意最大的两块——贡献质量（27）和生态影响力（20）——恰恰是最难造假的那两件。star（18）和粉丝（8）这两个能买的加起来，还不如光是合并 PR 值钱。这个排序，就是重点所在。

### 分量最重的信号：谁的代码，进了谁的仓库

单项分值最高的是**生态影响力**（20 分），值得专门讲讲它为什么最聪明。

它统计的是那些实质性的 Pull Request——改动超过五行、不是修错别字——合并进了**你并不拥有的热门仓库**。想象一个开发者，他真正的活儿是在某个知名项目的代码库里，而不是在自己那几个带 star 的仓库里。这个你造不了假。把一个真实改动合并进一个五万 star 的项目，意味着一个没理由帮你的维护者看过你的代码、点了头。这是 GitHub 上最接近"同行评审过的资历证明"的东西。

这里有一个刻意留的例外。如果那个热门仓库是**你自己的**——但确实很热门，1000 star 以上——那也算，因为它要照顾到那种把时间花在做自己知名项目、而不是给别人贡献的创造者。**不**算的，是往你自己那些没人 star 的小仓库里提 PR。往一个你昨天刚建、没人理的项目里反复提 PR，是刷贡献数的经典手法，所以在这里被排除掉（并在别处被扣分）。

## 为什么大数字不会一家独大

一个天真的算法，会让某一个爆红的仓库、或者某个有十万粉丝的账号，把一切都盖过去。这套算法不会，原因就是一个设计选择：每一个"有多少"的数字，在变成分数之前，都要先过一道**边际递减的曲线**。

![边际递减曲线：得分对 star 数](/blog/how-we-score-github-accounts/diminishing-returns.svg "分数在几千 star 之前涨得很快，之后就趋平——所以一个爆红仓库或买来的 star 都无法一家独大。")

说人话：从 0 涨到 1000 个 star，能给你挣不少分；从 5 万涨到 5.1 万，几乎挣不到——你早就到顶了。这道曲线奖励你跨过一个有意义的门槛，但不让少数几个超大数字把其他一切都挤掉。一个有几千 star、又有稳定历史的扎实开发者，不会被某个人一个爆红仓库压在底下。它同时意味着买 star 的价值急剧递减：一开始买的那点几乎没用，而顺着曲线往上买，很快就贵得离谱、却几乎换不到分。

## 风险信号：抓造假的

在六个正向维度之上，引擎还会针对一些具体的、众所周知的作弊和低投入套路扣分。这些是机器人、垃圾号和刷量号的指纹。用大白话说几个主要的：

- **模板化 PR 灌水**——几十个几乎一模一样、自动生成的 Pull Request，通常都冲着同一个仓库去。这是刷出来的贡献史最强的破绽。
- **微小 PR 刷量**——一堆只改一行的"修个错别字"式 PR，把贡献数撑起来，却没有任何真活。
- **自我 PR 刷量**——往自己那些零 star 的仓库里提、再自己合并 PR 来堆数字。合并自己的代码什么都证明不了。
- **互关刷粉**——关注上千个账号来钓回关，留下一个严重失衡的关注/被关注比。
- **新号海量仓库**——一个上个月才建、却有五十个仓库的账号，几乎不可能是真开发者。
- **幽灵主页**——没简介、几乎没粉丝、没 star、几乎没有合并过的活。不是恶意，只是空。
- **疑似 star 注水**——一个 star 很多、但几乎没有 fork 和 issue 的仓库，这正是买来的 star 该有的样子。

这些扣分会累加（有上限），所以一个踩中好几条的账号，不管原始数字多好看，都会掉到底部附近。关键在于：这些套路存在于账号**历史**的层面，而不是任何单个动作——单独一个改一行的 PR 完全正常；一百个冲着同一个仓库去的，就不正常了。

## 最后这个数字意味着什么

把六个维度加起来，减掉风险扣分，你就落进四档里的一档：

| 分数 | 段位 | 含义 |
|---|---|---|
| 90–100 | **夯** | 顶级开发者——高价值、高信任。 |
| 70–89 | **人上人** | 优质贡献者——值得信任。 |
| 40–69 | **NPC** | 普通账号——特征平庸或存疑。 |
| 0–39 | **拉完了** | 低价值——大概率不活跃、空号或刷量。 |

段位名字故意起得有点皮——这东西一开始就是个毒舌打分工具——但名字背后的分档，对所有人都是同一套确定性的算法。

## 一点老实话：这个分数**不是**什么

- **它只看得到公开活动。** 一个在公司私有仓库里干得漂亮的人，在这里可能显得很单薄。低分说的是**公开**足迹，不是对这个人的判决。
- **它是起点，不是法官。** 这个数字是用来帮一个人排优先级的——先看哪个陌生人的 PR、哪个主页值得细读——而不是用来自动拒绝任何人的。分数背后的证据，比分数本身更重要。
- **近期行为比远古历史更算数。** 生态影响力这个信号看的是近期的 PR，所以一个大贡献全在好几年前的人，得分会低于他简历给人的印象。这是故意的：它量的是你**现在**在做什么。

## 它是开源的——你可以自己跑

以上这些都不是黑箱，而这正是重点。回路里没有模型，没有藏起来的权重，没有"信我就好"。相同的输入永远得到相同的分数，而上面描述的每一条规则——每个权重、每个阈值、每个风险信号的触发条件——都以 AGPL 协议公开。

- **读代码：** [github.com/hikariming/ghfind](https://github.com/hikariming/ghfind)
- **装引擎：** `npm install ghfind` 或 `pip install ghfind`
- **本地跑**，带上你自己的 GitHub token——什么都不会离开你的机器——或者调用公开 API（[OpenAPI 规范](https://ghfind.com/openapi.json)）。
- **给单个账号打分**，直接在浏览器里打开 [ghfind.com](https://ghfind.com)。

如果你不同意某个权重或某个阈值，你可以精确地读到它是什么、改掉它、看看效果。一个别人没法检查的信任分，值不了几个钱——所以我们做了一个你可以检查的。
