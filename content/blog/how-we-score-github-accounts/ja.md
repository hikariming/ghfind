**一言で言うと:** このスコアは、たった一つの実用的な問いに答えます——*このアカウントは本物で価値のある開発者なのか、それとも開発者らしく見せかけたものなのか？* そしてその答えを、毎回同じルールで、公開データだけを使い、すべてのルールを明るみに出したうえで返します。この記事では専門用語を使わず、その数字がどう作られるのかを正確に説明します。

## そもそもなぜスコアなのか

ますます多くの判断が、誰かの GitHub をちらっと見ることに頼るようになっています。採用担当者は通話の前にプロフィールに目を通す。メンテナは見知らぬ人の Pull Request をレビューする価値があるか判断する。ディレクトリ系のサイトは「どれだけ立派に見えるか」でアカウントを並べる。こうした用途のどれもが、その信号を*偽装する*動機を生みます——そして安い信号は偽装が簡単です。star は買える。フォロワーは交換できる。一行だけの Pull Request を午後いっぱいで百個開けば、自称「オープンソース貢献者」の出来上がりです。

だから役に立つスコアは、大きくて派手な数字を足し合わせるだけではダメです。本当に偽装しにくいものに重く寄りかかり、偽装コストの低いものは軽く受け流さなければなりません。以下のすべての設計は、この一つの考えから導かれています。

## 唯一の原則：偽装しにくいものに重い点を

GitHub のあらゆる信号を、2つのバケツに分けます。

- **偽装が安い：** star、フォロワー。数ドルか、フォロー返しの輪があれば作れます。
- **偽装が高い：** **他人**が管理する本物のプロジェクトにマージされた Pull Request、何年も続く安定した活動、多忙なメンテナが実際に受け入れたコード。

エンジンは2つ目のバケツに重く加点し、1つ目には軽く加点します。star とフォロワーもちゃんと点になります——本当に人気のあるプロジェクトは*当然*あなたの助けになるべきです——が、その上限は十分低く抑えてあり、買ってもほとんど針は動きません。一方で、本物のコードを著名なリポジトリにマージすること、つまりあなたを助ける理由のない人間を説得することは、盤面全体で最も点が高い行為です。

哲学はこれだけです。あとは、それを6つのカテゴリーにどう配分するかにすぎません。

## 測る6つのこと

スコアは 0 から 100 まで、6つの次元に配分されます。以下、それぞれをわかりやすく、満点付きで。

| 次元 | 満点 | 本当に問うていること |
|---|---|---|
| **貢献の質** | 27 | 本物の PR を本物のプロジェクトにマージできているか、メンテナはそれを受け入れているか？ |
| **エコシステムへの影響** | 20 | あなたのコードは、本当に人気のある——しかも自分のものではない——リポジトリに入ったか？ |
| **オリジナルプロジェクトの質** | 18 | 人が実際に使うものを作ったか（star で測るが、上限あり）？ |
| **活動の真正性** | 17 | 長期にわたり多様に安定して活動しているか、それとも一度爆発して沈黙したか？ |
| **アカウントの成熟度** | 10 | このアカウントはどれだけ長く存在し、活動してきたか？ |
| **コミュニティ影響力** | 8 | 本物のフォロワーがいて、その比率は健全か？ |

![100点が次元ごとにどこへ配分されるか](/blog/how-we-score-github-accounts/weight-breakdown.svg "6つの次元とその満点。オレンジは偽装しにくい信号、グレーは買える信号。")

最大の2つ——貢献の質（27）とエコシステムへの影響（20）——が、まさに最も偽装しにくいものだと気づいてください。買える信号である star（18）とフォロワー（8）を足しても、マージされた PR 単体にすら及びません。この順序こそが要点です。

### 最も重い信号：誰のコードが、誰のリポジトリに

単項目で最も点が高いのは**エコシステムへの影響**（20点）で、これがなぜ最も巧妙なのか、専用に説明する価値があります。

これが数えるのは、実質的な Pull Request——五行を超える変更で、誤字修正ではないもの——が、**あなたが所有していない人気リポジトリ**にマージされた回数です。ある開発者を思い浮かべてください。彼の本当の仕事は、自分の star 付きリポジトリではなく、ある著名プロジェクトのコードベースの中にある。これは偽装できません。本物の変更を5万 star のプロジェクトにマージするというのは、あなたを助ける動機のないメンテナがあなたのコードを見て、うなずいた、ということです。これは GitHub において「査読を経た資格」に最も近いものです。

ここには意図的な例外が一つあります。その人気リポジトリが**あなた自身のもの**であっても——ただし本当に人気で、1000 star 以上なら——それも数えます。自分の著名プロジェクトを作ることに時間を使う制作者を拾うためです。**数えない**のは、誰も star を付けない自分の小さなリポジトリへの PR です。昨日作って誰も見向きもしないプロジェクトに繰り返し PR を開くのは、貢献数を水増しする典型的な手口なので、ここでは除外されます（そして別の場所でペナルティを受けます）。

## なぜ大きな数字が独走しないのか

素朴なスコアなら、たった一つのバズったリポジトリや、10万フォロワーのアカウントが、すべてを覆い尽くしてしまいます。このスコアはそうなりません。理由は一つの設計選択です。すべての「どれくらい」の数字は、点数になる前に**逓減の曲線**を通ります。

![逓減曲線：獲得点数と star 数](/blog/how-we-score-github-accounts/diminishing-returns.svg "点数は数千 star までは速く伸び、その後は平らになる——だからバズったリポジトリや買った star が独走することはない。")

わかりやすく言えば、0 から 1000 star へ伸びればかなりの点が入りますが、5万から5.1万へ伸びてもほとんど入りません——すでに頂点近くにいたからです。この曲線は、意味のある基準を越えたことには報いつつ、少数の巨大な数字が他のすべてを押しのけることは許しません。数千 star と安定した履歴を持つ堅実な開発者が、誰かのバズった一つのリポジトリの下敷きにされることはありません。同時に、star 買いの価値は急激に逓減します。最初に買った分はほとんど効かず、曲線を登るように買い進めると、すぐに法外に高くつくのに見返りはほぼゼロです。

## リスク信号：偽物を捕まえる

6つのプラスの次元に加えて、エンジンは具体的でよく知られた不正・手抜きのパターンに対して減点します。これらはボット、スパム、水増しアカウントの指紋です。主なものを、わかりやすく数えます。

- **テンプレート PR の氾濫**——ほとんど同一の自動生成された Pull Request が何十個も、たいてい同じリポジトリを狙う。水増しされた貢献履歴の最強の兆候です。
- **些末な PR の量産**——一行だけの「誤字修正」PR の山で、実質のない貢献数だけを膨らませる。
- **自己 PR の量産**——自分の star ゼロのリポジトリに自分で PR を開いてマージし、数字を盛る。自分のコードをマージしても何も証明できません。
- **フォロー狩り**——数千のアカウントをフォローしてフォロー返しを釣り、フォロー/フォロワー比を極端に歪める。
- **新規アカウントの大量リポジトリ**——先月作ったのに50個のリポジトリがあるアカウントは、まず本物の開発者ではありません。
- **ゴーストプロフィール**——bio なし、フォロワーほぼゼロ、star なし、マージされた仕事もほぼなし。悪意ではなく、ただ空っぽ。
- **star 水増しの疑い**——star は多いのに fork も issue もほとんどないリポジトリ。買った star はまさにこう見えます。

これらの減点は（上限付きで）積み重なるので、複数に引っかかったアカウントは、原数字がどれだけ立派でも底辺付近に落ちます。肝心なのは、これらのパターンはアカウントの*履歴*のレベルに存在し、単一の行動のレベルにはないということです——一行の PR が単独であるのは完全に正常で、同じリポジトリを狙った百個は正常ではありません。

## 最後の数字が意味するもの

6つの次元を足し、リスク減点を引くと、4つの段位のどれかに落ち着きます。

| スコア | 段位 | 意味 |
|---|---|---|
| 90–100 | **夯（トップ級）** | 一流の開発者——高価値・高信頼。 |
| 70–89 | **人上人（傑出）** | 良質な貢献者——信頼に値する。 |
| 40–69 | **NPC** | 平凡なアカウント——特徴が凡庸か、判断がつかない。 |
| 0–39 | **拉完了（お手上げ）** | 低価値——おそらく非活動、空っぽ、または水増し。 |

段位の名前はわざと少しふざけています——もともと毒舌採点ツールとして始まったので——が、その背後の区分は、誰に対しても同じ決定論的な計算です。

## 正直に言う：このスコアが**そうでない**もの

- **公開された活動しか見えません。** 会社のプライベートリポジトリで素晴らしい仕事をしている人は、ここでは薄く見えることがあります。低スコアは*公開された*足跡についての言明であって、その人への判決ではありません。
- **出発点であって、裁判官ではありません。** この数字は人が優先順位をつけるのを助けるためのもの——どの見知らぬ人の PR を先に見るか、どのプロフィールを丁寧に読むか——であって、誰かを自動で却下するためのものではありません。スコアの背後にある証拠のほうが、スコア自体より重要です。
- **最近の行動が、大昔の履歴より重い。** エコシステムへの影響の信号は最近の PR を見るので、大きな貢献がすべて何年も前という人は、経歴書の印象より低く出ます。これは意図的で、あなたが*今*何をしているかを測っています。

## オープンソースです——自分で動かせます

これらはどれもブラックボックスではなく、そこが要点です。ループの中にモデルはなく、隠れた重み付けもなく、「信じてくれ」もありません。同じ入力は常に同じスコアを生み、上で説明したすべてのルール——あらゆる重み、あらゆる閾値、あらゆるリスク信号の発動条件——は AGPL ライセンスで公開されています。

- **コードを読む：** [github.com/hikariming/ghfind](https://github.com/hikariming/ghfind)
- **エンジンを入れる：** `npm install ghfind` または `pip install ghfind`
- **ローカルで動かす**——自分の GitHub トークンを使えば、何もマシンの外に出ません——または公開 API を呼ぶ（[OpenAPI 仕様](https://ghfind.com/openapi.json)）。
- **単一アカウントを採点する**なら、ブラウザで [ghfind.com](https://ghfind.com) を開くだけ。

もしある重みや閾値に納得がいかなければ、それが正確に何なのかを読み、書き換え、効果を見ることができます。誰も検査できない信頼スコアには大した価値がありません——だから私たちは、検査できるものを作りました。
