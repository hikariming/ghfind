**초록.** GitHub 상의 조작된 활동 — 템플릿화된 풀 리퀘스트(PR), 구매한 스타, 맞팔로우 네트워크 — 에 대한 우려는 널리 퍼져 있으나, 그 비율과 구조에 관한 정량적 증거는 드뭅니다. 본 연구에서는 결정론적 오픈소스 평가 기준(모델 호출 없음, 동일한 입력은 동일한 점수를 산출)으로 18,947개의 공개 GitHub 계정을 점수화하고, 그중 3,444개 계정에 대해 완전한 원시 지표 스냅숏을 보존한 심층 샘플을 분석합니다. 주요 발견은 다음과 같습니다. (i) 본 샘플에서 조작은 드뭅니다 — 보수적으로 설정한 파밍 임계값을 초과하는 계정은 0.58%에 불과합니다. (ii) 조작이 나타날 때는 극단적이며 정상 행동과 쌍봉형으로 분리됩니다 — 플래그된 계정의 템플릿화 PR 제목 비율은 모두 50%를 넘는 반면(최대 97%), 전체 모집단의 중앙값은 7%입니다. (iii) 저비용 기여는 의심스러운 것이 아니라 정상입니다 — 최근 병합된 PR 샘플이 있는 개발자의 58%가 최소 한 건의 사소한 외부 PR을 보유한 반면, 사소한 PR이 대량으로 과반을 차지하는 패턴을 보이는 비율은 0.4%에 그칩니다. (iv) "의심스러워 보이는" 프로필의 지배적 형태는 조작이 아니라 공백입니다 — 부재형 위험 신호가 조작형 신호보다 한 자릿수 더 많습니다. 중앙값 점수는 계정 연령에 따라 단조 증가하여 1년 미만의 18점에서 10년 초과의 86점에 이르며, 이는 장기간의 일관성이 조작하기 가장 어려운 신호라는 가설과 부합합니다. 이를 토대로 스팸 탐지 도구 설계에 대한 시사점을 논의하며, 특히 파밍은 개별 기여 수준의 사건 속성이 아니라 계정 이력 수준의 패턴 속성이라는 점을 강조합니다.

## 1. 서론

개발자 신뢰도에 대한 평가는 점점 더 공개 GitHub 활동에 의존하고 있습니다. 채용 파이프라인은 후보자의 프로필을 심사하고, 오픈소스 메인테이너는 낯선 기여자의 풀 리퀘스트를 분류하며, 하위 도구들은 외견상의 영향력에 따라 계정의 순위를 매깁니다. 이러한 용도 하나하나가 그 기반이 되는 신호를 조작할 유인을 만들어냅니다. 스타 거래 시장, 템플릿화된 풀 리퀘스트 공세, 맞팔로우 수법에 관한 일화적 보고는 흔하지만, 그러한 조작이 얼마나 자주 발생하며 통계적으로 어떤 형태를 띠는지에 대한 체계적 측정은 드뭅니다.

본 데이터셋의 한 사례가 이 현상을 잘 보여 줍니다. 어떤 계정의 병합된 PR 기록은 통상적 기준으로는 강력한 기여자임을 시사합니다. 병합된(단순히 열린 것이 아닌) 풀 리퀘스트가 다수이고, 수락률이 거의 완벽합니다. 그러나 자세히 살펴보면 최근 PR 제목의 97%가 거의 동일한 템플릿 변형이며, 그 대다수가 해당 계정이 소유하지 않은 단일 인기 저장소를 대상으로 하고 있습니다. 어떤 풀 리퀘스트도 개별적으로는 이상하지 않으며, 이상은 오직 집계된 패턴의 수준에서만 존재합니다. 조작이 사건 단위로는 보이지 않지만 집계 수준에서는 뚜렷하다는 이 관찰이 본 연구의 출발점입니다.

우리는 세 가지 질문을 제기합니다:

1. **비율.** 공개 GitHub 계정 가운데 조작된 기여 활동은 얼마나 흔한가?
2. **구조.** 조작이 발생할 때, 그것은 통계적으로 정상적인 기여 행동과 어떻게 다른가?
3. **구성.** 무결성 휴리스틱에 걸린 계정 가운데, 능동적 조작을 반영하는 비율은 얼마이고 단순한 비활성 내지 독창적 작업의 부재를 반영하는 비율은 얼마인가?

이에 답하기 위해, 우리는 결정론적 평가 기준([ghfind](https://ghfind.com), 점수화 코어는 AGPL 라이선스로 오픈소스 공개, [저장소](https://github.com/hikariming/ghfind))으로 18,947개의 공개 계정을 점수화하고, PR 수준 샘플, 저장소 품질 특성, 활동 형태 통계를 포함한 완전한 원시 지표 스냅숏이 보존된 3,444개 계정의 심층 샘플을 분석합니다. 각 그림의 기반이 되는 집계 데이터는 본 논문과 함께 공개됩니다([data.json](/blog/we-scored-19000-github-accounts/data.json)).

요약하면, 본 샘플에서 조작은 공적 담론이 시사하는 것보다 훨씬 드물고, 존재할 때는 미묘하기보다 극단적이며, 단순한 패턴 수준의 임계값만으로도 정상 활동과 분리할 수 있습니다.

## 2. 데이터와 방법론

### 2.1 평가 기준

엔진은 합계 100점을 이루는 여섯 개 차원의 결정론적 평가 기준을 구현하며, 위험 신호에 대해 가산식 감점을 적용합니다. 모델 호출은 수행하지 않으며, 점수는 공개 GitHub 데이터로부터 완전히 재현 가능합니다. ghfind 웹사이트, npm/PyPI SDK, 그리고 본 분석에 사용된 점수는 동일한 코드 경로에서 산출됩니다.

| 차원 | 만점 | 보상하는 신호 |
|---|---|---|
| 기여 품질 | 27 | 병합된 PR 수(로그 스케일), 수락률, 이슈 참여 |
| 생태계 영향력 | 20 | 높은 스타 저장소에 대한 실질적 PR, 메인테이너로서의 깊이 |
| 독창적 프로젝트 품질 | 18 | 저장소의 실질성으로 가중된 스타 수 |
| 활동 진정성 | 17 | 지속적인 최근 활동, 활동 유형의 다양성 |
| 계정 성숙도 | 10 | 계정 연령, 실제 활동 연수 |
| 커뮤니티 영향력 | 8 | 팔로워 수(로그 스케일), 팔로워/팔로잉 비율의 타당성 |

`templated_pr_flooding`(템플릿화 PR 플러딩), `trivial_pr_farming`(사소한 PR 파밍), `follow_farming`(팔로우 파밍), `possible_star_inflation`(스타 부풀리기 의심)을 포함한 열두 개의 결정론적 위험 신호 규칙이 감점을 담당합니다. 정확한 임계값은 저장소에서 확인할 수 있습니다. 공개 점수 외에도 엔진은 리더보드의 무결성을 보호하기 위해 0–10 스케일의 내부 스팸/봇 가능성 점수를 계산하며, 3.2절에서 그 분포를 처음으로 공개합니다. 그 밖의 비공개 데이터는 본 분석에 사용되지 않습니다.

**정의.** 변경이 5행 이하이고, 작성자가 소유하지 않은 스타 200개 이상의 저장소에 병합된 풀 리퀘스트를 **사소한(trivial) PR**이라 부릅니다. 한 계정의 **템플릿화 제목 비율**은 최근 PR 제목 가운데 서로 거의 동일한 템플릿 변형이 차지하는 비율을 뜻합니다.

### 2.2 샘플 구성과 알려진 편향

샘플은 (a) ghfind 웹사이트를 통해 자발적으로 자신의 계정을 점수화한 사용자와 (b) 활발한 오픈소스 조직으로부터 수집된 개발자로 구성됩니다. 이 설계의 두 가지 속성이 해석을 제약합니다. 첫째, 샘플은 자기선택적이며 진정성 있고 활동적인 개발자 쪽으로 치우쳐 있습니다. 따라서 아래에 보고되는 모든 조작 비율은 GitHub 전체에 대한 추정치가 아니라 **이미 필터링된 모집단 내에서의 하한**으로 읽어야 합니다. 둘째, 점수화된 계정 18,947개(심층 지표 보유 3,444개)는 분포의 형태를 특징짓기에는 충분히 크지만 GitHub 전체에 비하면 무시할 만한 비율입니다. 우리가 보고하는 것은 형태이지, 전수 조사가 아닙니다.

## 3. 결과

### 3.1 점수 분포

![19k 계정의 최종 점수 분포](/blog/we-scored-19000-github-accounts/score-distribution.svg "그림 1: 5점 구간별 최종 점수 분포(n = 18,947). 주황색 띠는 70점 이상 구간을 표시합니다.")

*그림 1*은 최종 점수의 분포를 보여 줍니다. 중앙값은 40점을 조금 넘으며, **48.6%**의 계정이 40점 미만(평가 기준이 저가치 또는 부풀리기 의심으로 분류하는 구간)이고, 90점을 초과하는 계정은 **3.7%**에 불과합니다. 가장 많은 계정이 몰린 구간은 0–5점으로, 독창적 작업도, 병합된 풀 리퀘스트도, 지속적 활동도 없는 계정들로 이루어져 있습니다. 활동적인 개발자 쪽으로 치우친 샘플에서조차 대부분의 공개 프로필은 빈약합니다.

기준점을 제시하면, 심층 샘플의 중앙값 계정은 **팔로워 27명, 총 스타 34개, 병합된 PR 20건**을 보유하며, 계정 연령의 중앙값은 7년입니다. 평판과 관련된 지표는 상위 꼬리에 크게 집중되어 있습니다. 90번째 백분위는 팔로워 1,275명과 약 5,900 스타이고, 99번째 백분위는 팔로워 19,000명과 약 100,000 스타입니다.

### 3.2 파밍의 비율과 구조

![숨겨진 스팸 점수 분포](/blog/we-scored-19000-github-accounts/spam-score.svg "그림 2: 내부 0–10 스팸 가능성 점수의 분포(n = 18,934). 77%의 계정이 정확히 0점입니다.")

*그림 2*는 해당 점수가 계산된 18,934개 계정에 대한 내부 스팸 가능성 점수의 분포를 보고합니다:

- 계정의 **77%**가 정확히 0점 — 어떤 종류의 파밍 신호도 없습니다.
- **0.58%**(110개 계정)가 3점 이상으로, 이는 우리가 명백한 파밍으로 간주하는 임계값입니다.
- **12개 계정**이 7점 이상입니다.

파밍 계정의 분리 가능성은 그 양이 아니라 *집중도*에서 나옵니다. 최근 풀 리퀘스트가 10건 이상인 2,122명의 개발자 가운데 템플릿화 제목 비율의 중앙값은 **7%**입니다 — 평범한 개발자도 어느 정도는 같은 제목을 반복합니다("fix typo", "bump deps"). 반면 템플릿 플러딩으로 플래그된 모든 계정의 비율은 **50%**를 넘어 **97%**까지 이르는데, 전체 모집단의 99번째 백분위는 72%입니다. 이 분포는 사실상 쌍봉형입니다. 평범한 반복과 일괄 생성 사이에 연속체는 존재하지 않습니다. 이러한 계정을 탐지하는 데 학습된 모델은 필요하지 않으며, 계정별로 두 건 이상의 풀 리퀘스트를 함께 집계하는 것으로 충분합니다.

### 3.3 사소한 풀 리퀘스트는 정상이지만, 사소한 PR 파밍은 아니다

최근 병합된 PR 샘플이 있는 2,558명의 개발자 가운데 **58%**가 2.1절에서 정의한 사소한 외부 PR을 최소 한 건 보유하고 있습니다. 유명 저장소에 대한 작은 문서 수정과 오타 수정은, 그 외의 면에서는 강력한 기여자를 포함하여, 오픈소스 참여의 일상적인 부분임이 분명합니다.

이와 대조적으로, 최근 병합된 PR의 *과반*이 사소하고 그러한 PR을 10건 이상 보유한 개발자는 샘플의 **0.4%**(11개 계정)에 해당합니다.

이 두 수치 사이의 간극은 어뷰징 방지 도구의 설계에 직접적인 함의를 갖습니다. 작은 풀 리퀘스트 자체에 벌점을 주는 휴리스틱은, 패턴 수준에서 이미 눈에 띄는 행동을 보인 11개 계정을 잡기 위해 그 58% — 신규 참여자, 문서 기여자, 지나가다 돕는 조력자 — 에게 벌점을 주게 됩니다. 우리는 파밍이 개별 diff의 **사건 속성**이 아니라 계정 기여 이력의 **패턴 속성**(집중도, 템플릿화, 단일 대상에 대한 반복)이며, 단일 사건에 대해 작동하는 탐지기는 잘못된 양을 측정하고 있다고 결론짓습니다.

거부율에 대해서도 평행한 결과가 성립합니다. 판정이 내려진 PR이 10건 이상인 2,003명의 개발자 가운데 메인테이너 거부율의 중앙값은 **2%**이고 90번째 백분위는 13%입니다. 따라서 수락률은 판별 신호를 거의 담고 있지 않으며 — 병합된 PR이 있기만 하다면, 그러한 기여자가 제출하는 것은 거의 모두 병합됩니다 — 극단적인 거부율(평가 기준은 50% 초과를 플래그)만이 유의미합니다.

### 3.4 위험 신호의 구성: 부재가 조작을 압도한다

현재 엔진을 3,444개 심층 샘플 계정 전체에 대해 다시 실행하면 **17%**가 최소 한 개의 위험 신호에 걸립니다. *그림 3*은 그 구성을 보여 줍니다.

![위험 신호 출현율](/blog/we-scored-19000-github-accounts/red-flags.svg "그림 3: 심층 샘플에서의 위험 신호 출현율(n = 3,444). 부재형 신호가 조작형 신호를 한 자릿수 차이로 상회합니다.")

출현율이 가장 높은 세 신호 — `ghost_profile`(10%), `no_original_work`(8.2%), `mostly_forks`(7.2%) — 는 조작이 아니라 실질의 부재를 기술합니다. 능동적 기만 신호는 한 자릿수 더 드뭅니다: `templated_pr_flooding` 0.5%, `trivial_pr_farming` 0.3%, `follow_farming` 0.1%. 스타 부풀리기 — 포크와 이슈가 거의 0인데 스타 수가 높은 경우 — 는 본 샘플에서 0으로 반올림되지만, 자기선택이 이를 억누르고 있을 개연성이 있습니다. 스타를 구매한 계정이 스스로 점수화를 신청할 가능성은 낮기 때문입니다.

GitHub 데이터 위에 구축되는 신뢰 도구에게 이 구성은, 서로 다른 탐지기와 서로 다른 오류 허용치를 요구하는, 질적으로 다른 두 가지 실패 양상을 시사합니다. 흔하고 비용이 낮은 경우("여기에는 아무것도 없다")와 드물고 비용이 큰 경우("여기 있는 무언가는 만들어진 것이다")입니다.

### 3.5 위조 불가능한 신호로서의 계정 연령

![계정 연령별 중앙값 점수](/blog/we-scored-19000-github-accounts/age-vs-score.svg "그림 4: 계정 연령별 중앙값 최종 점수. 이 관계는 모든 연령 구간에 걸쳐 단조입니다.")

중앙값 점수는 계정 연령에 따라 단조 증가하며, **1년 미만 계정의 18점에서 10년을 넘긴 계정의 86점**까지 중간의 하락 없이 상승합니다(*그림 4*). 생존 편향이 이 관계에 기여하지만 — 샘플에 나타나는 오래된 계정은 여전히 사용 중인 오래된 계정입니다 — 효과의 방향 자체가 정보를 담고 있습니다. 장기 일관성의 모든 구성 요소(활동 연수, 스타가 유기적으로 축적된 오래된 저장소, 여러 릴리스에 걸친 기여 이력)는 정확히 조작이 압축할 수 없는 것입니다. 스타와 팔로워는 몇 시간 안에 획득할 수 있지만, 9년의 활동 이력을 지닌 2015년 계정을 2026년에 새로 만들어 낼 수는 없습니다. 이 비대칭성은 평가 기준이 계정 성숙도와 활동 기간에 부여한 가중치를 정당화하며, 플래그된 플러딩 계정이 젊은 계정에 몰려 있다는 우리의 관찰과도 부합합니다.

부차적 관찰로, 고득점 계정(최종 점수 ≥ 60)의 주 언어 구성은 익숙한 생태계 추세를 따릅니다. **TypeScript(520), Python(460), JavaScript(395)**가 선두이고, Rust(225)가 Go(189), C(184), Java(159)를 앞섭니다(*그림 5*).

![고득점자의 언어](/blog/we-scored-19000-github-accounts/languages.svg "그림 5: 60점 이상을 기록한 계정들의 주 언어.")

## 4. 논의

이상의 결과로부터 기여 무결성 도구를 위한 세 가지 설계 원칙이 도출됩니다.

**사건이 아니라 패턴을 탐지하라.** 3.2절의 쌍봉형 분리와 3.3절의 58% 대 0.4% 간극은, 개별 기여는 조작 신호를 거의 담고 있지 않은 반면 계정 수준의 집계는 깨끗하게 분리됨을 보여 줍니다. 단일 사건 휴리스틱은 오픈소스 프로젝트가 가장 붙들어야 할 기여자들 — 신규 참여자와 저빈도 조력자 — 에 대해 오탐을 극대화합니다.

**부재와 조작을 구분하라.** 무결성 검사에 걸리는 계정 대부분은 적대적인 것이 아니라 비어 있는 것입니다(3.4절). 이 둘을 혼동하면 외견상의 사기율이 부풀려지고 검토 노력이 잘못 배분됩니다.

**시간에 가중치를 부여하라.** 종단적 일관성은 조사된 신호 가운데 비용 지출로 획득 비용을 줄일 수 없는 유일한 신호이며(3.5절), 따라서 모든 신뢰도 점수의 자연스러운 닻이 됩니다.

이 원칙들에 따라, 우리는 PR 수준 특성(diff 크기, 템플릿 유사도)과 작성자 수준 이력을 결합하여 메인테이너의 스팸 풀 리퀘스트 분류를 돕는 GitHub App을 개발하고 있습니다. 3.3절에서 기록한 오탐 위험을 고려하여, 이 도구는 풀 리퀘스트를 자동으로 닫지 않으며 사람의 결정을 위한 증거를 제시합니다. 피해를 겪는 저장소의 메인테이너들께서 받고 계신 스팸의 사례를 공유해 주시기를 청합니다.

## 5. 한계

- **자기선택 샘플.** 웹사이트 사용자와 활발한 오픈소스 조직의 구성원은 진정성 있고 활동적인 쪽으로 치우칩니다. GitHub 전체의 조작 비율은 여기 보고된 것보다 높을 개연성이 있으며, 본 논문의 백분율은 필터링된 모집단 내에서의 하한입니다.
- **규모.** 점수화된 계정 18,947개, 심층 지표 보유 3,444개. 분포의 형태는 재실행 간에 안정적이지만, 샘플이 GitHub 전체를 대표하지는 않습니다.
- **집계 보고만 수행.** 모든 패턴은 집계 형태로만 보고되며, 어떤 개별 계정도 특정되지 않습니다. 플래그 임계값은 메인테이너가 맥락과 함께 적용할 수 있도록 공개되어 있습니다.
- **엔진 버전.** 위험 신호 출현율은 각 계정의 최신 원시 스냅숏에 대해 현재 평가기로 재계산했으며, 저장된 스팸 점수는 스캔 시점의 엔진 버전을 반영합니다. 둘 다 [공개된 집계 데이터](/blog/we-scored-19000-github-accounts/data.json)에 포함되어 있습니다.

## 6. 재현 가능성

모든 점수화 로직은 결정론적이며 [github.com/hikariming/ghfind](https://github.com/hikariming/ghfind)에서 AGPL 라이선스로 오픈소스 공개되어 있습니다. 동일한 엔진이 `npm install ghfind`와 `pip install ghfind`로 배포되며, 공개 API([OpenAPI 명세](https://ghfind.com/openapi.json))를 상대로 실행할 수도 있고, 사용자가 제공한 GitHub 토큰으로 완전히 로컬에서 실행할 수도 있습니다. 본 논문의 모든 그림 뒤에 있는 집계 통계는 [data.json](/blog/we-scored-19000-github-accounts/data.json)으로 제공됩니다.
