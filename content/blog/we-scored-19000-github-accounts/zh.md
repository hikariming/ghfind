**摘要。** 关于 GitHub 上存在伪造活动（模板化 Pull Request、购买 star、互关网络等）的担忧十分普遍，但针对其流行度与统计结构的定量证据却很少。我们使用一套确定性、开源的评分规则（不调用任何模型；相同输入产生相同分数）对 18,947 个公开 GitHub 账号进行评分，并对其中 3,444 个账号保留完整的原始指标快照作为深度样本。主要发现如下：(i) 在本样本中，造假行为罕见——仅 0.58% 的账号超过我们保守设定的刷量阈值；(ii) 造假一旦出现即表现为极端形态，与正常行为呈双峰分离——被标记账号的模板化 PR 标题占比全部高于 50%（最高 97%），而总体中位数仅为 7%；(iii) 低投入贡献是常态而非可疑信号——在有近期合并 PR 样本的开发者中，58% 至少有一个琐碎的外部 PR，而呈现规模化"以琐碎为主"模式的仅占 0.4%；(iv) 触发完整性启发式规则的账号中，占主导的形态是"空缺"而非"操纵"——空缺类红旗的流行度比操纵类红旗高一个数量级。账号中位数分数随账号年龄单调上升，从不满一年的 18 分升至十年以上的 86 分，与"长周期一致性是最难伪造的信号"这一假设相符。我们据此讨论了反垃圾工具的设计启示，核心结论是：刷量是账号历史层面的模式属性，而非单个贡献层面的事件属性。

## 1. 引言

对开发者可信度的评估正日益依赖公开的 GitHub 活动：招聘流程筛查候选人档案，开源维护者对陌生贡献者的 Pull Request 进行分诊，下游工具按表观影响力对账号排序。每一种用途都构成伪造底层信号的激励。关于 star 交易市场、模板化 PR 攻势和互关刷粉的轶事性报告很常见；但对此类造假发生频率及其统计形态的系统性测量并不多见。

数据集中的一个例子可以说明这一现象。某账号呈现出通常意味着强贡献者的合并 PR 记录：大量被合并（而非仅提交）的 Pull Request，且通过率接近完美。细察发现，其近期 PR 标题中 97% 是几乎相同的模板变体，且多数指向同一个该账号并不拥有的热门仓库。没有任何单个 PR 是异常的；异常只存在于聚合模式的层面。这一观察——造假在逐事件层面不可见、在聚合层面却显著——构成了本研究的动机。

我们提出三个问题：

1. **流行度。** 在公开 GitHub 账号中，伪造贡献活动有多常见？
2. **结构。** 造假发生时，其统计特征与正常贡献行为有何差异？
3. **构成。** 在触发完整性启发式规则的账号中，主动操纵与单纯的不活跃或缺乏原创工作各占多少？

为回答这些问题，我们使用一套确定性规则（[ghfind](https://ghfind.com)，评分核心以 AGPL 协议开源，见[代码仓库](https://github.com/hikariming/ghfind)）对 18,947 个公开账号评分，并分析了保留完整原始指标快照（含 PR 级样本、仓库质量特征与活跃形态统计）的 3,444 个账号构成的深度样本。各图背后的聚合数据与本文一并发布（[data.json](/blog/we-scored-19000-github-accounts/data.json)）。

主要结论：在本样本中，造假远比公共讨论所暗示的罕见；一旦出现即为极端形态而非渐变；且仅凭模式层面的特征即可与真实活动近乎线性可分。

## 2. 数据与方法

### 2.1 评分规则

引擎实现了一套覆盖六个维度、合计 100 分的确定性规则，并对红旗信号施加累加式扣分。引擎不调用任何模型，分数可完全由公开 GitHub 数据复现。ghfind 网站、npm/PyPI SDK 与本文分析使用同一套代码路径。

| 维度 | 满分 | 奖励的信号 |
|---|---|---|
| 贡献质量 | 27 | 合并 PR 数（对数刻度）、通过率、issue 参与 |
| 生态影响力 | 20 | 对高星仓库的实质性 PR、维护者深度 |
| 原创项目质量 | 18 | 以仓库实质加权的 star 数 |
| 活跃真实性 | 17 | 持续的近期活跃、活动类型多样性 |
| 账号成熟度 | 10 | 账号年龄、实际活跃年数 |
| 社区影响力 | 8 | 关注者数（对数刻度）、关注/被关注比的合理性 |

十二条确定性红旗规则负责扣分，包括 `templated_pr_flooding`（模板化 PR 洪泛）、`trivial_pr_farming`（琐碎 PR 刷量）、`follow_farming`（互关刷粉）、`possible_star_inflation`（疑似 star 注水）等。具体阈值见代码仓库。在公开分数之外，引擎还计算一个 0–10 的内部垃圾/机器人可能性分数，用于保护榜单完整性；第 3.2 节首次公布其分布。除此之外，本分析不使用任何非公开数据。

**定义。** 若一个 Pull Request 改动不超过五行、且被合并进作者并不拥有的 200 star 以上仓库，我们称其为**琐碎 PR**。一个账号的**模板化标题占比**指其近期 PR 标题中互为近似模板变体的比例。

### 2.2 样本构成与已知偏差

样本由两部分构成：(a) 通过 ghfind 网站主动为自己账号评分的用户；(b) 从活跃开源组织收录的开发者。这一设计的两个性质约束了结论的解释范围。第一，样本为自选择样本，偏向真实、活跃的开发者；因此下文报告的所有造假比例均应理解为**已过滤人群内的下限**，而非 GitHub 全站估计。第二，18,947 个已评分账号（其中 3,444 个含深度指标）足以刻画分布形状，但相对 GitHub 总量可忽略不计；我们报告的是形状，不是普查。

## 3. 结果

### 3.1 分数分布

![19k 账号的最终分数分布](/blog/we-scored-19000-github-accounts/score-distribution.svg "图 1：最终分数分布，5 分一桶（n = 18,947）。橙色标记 70 分以上段位。")

**图 1** 展示了最终分数的分布。中位数略高于 40 分；**48.6%** 的账号低于 40 分（规则将该档标注为低价值/疑似刷量），仅 **3.7%** 超过 90 分。人数最多的单个分数桶为 0–5 分，由没有原创工作、没有合并 PR、没有持续活跃的账号构成。即使在偏向活跃开发者的样本中，多数公开档案依然单薄。

作为参照基准：深度样本中的中位数账号拥有 **27 个关注者、34 个 star、20 个合并 PR**，中位账号年龄为七年。与声誉相关的指标高度集中于上尾：第 90 百分位为 1,275 关注者、约 5,900 star；第 99 百分位为 19,000 关注者、约 100,000 star。

### 3.2 刷量行为的流行度与结构

![隐藏反刷分分布](/blog/we-scored-19000-github-accounts/spam-score.svg "图 2：内部 0–10 垃圾可能性分数的分布（n = 18,934）。77% 的账号得分恰为 0。")

**图 2** 报告了 18,934 个计算过该分数的账号上内部垃圾可能性分数的分布：

- **77%** 的账号得分恰为 0——不存在任何刷量信号。
- **0.58%**（110 个账号）达到 3 分及以上，即我们视为明确刷量的阈值。
- **12 个账号**达到 7 分及以上。

刷量账号的可分性并非来自其数量，而是来自其**集中度**。在近期拥有至少 10 个 PR 的 2,122 名开发者中，模板化标题占比的中位数为 **7%**——普通开发者存在适度的自我重复（"fix typo"、"bump deps"）。而所有因模板洪泛被标记的账号占比均高于 **50%**，最高达 **97%**；作为对比，全体人群的第 99 百分位仅为 72%。该分布实际上呈双峰形态：在"正常重复"与"批量生成"之间不存在连续过渡。检测这类账号不需要学习模型，只需要在单个账号内对多于一个 PR 做聚合。

### 3.3 琐碎 PR 是常态，琐碎 PR 刷量不是

在拥有近期合并 PR 样本的 2,558 名开发者中，**58%** 至少有一个符合 2.1 节定义的琐碎外部 PR。可见，向知名仓库提交小型文档或拼写修复是开源参与的常规组成部分，对原本就很强的贡献者亦然。

与之相对，近期合并 PR **以琐碎为主**且此类 PR 达到 10 个以上的开发者仅占样本的 **0.4%**（11 个账号）。

这两个数字之间的差距对反滥用工具设计有直接影响。任何对"小 PR"本身施加惩罚的启发式规则，都会为了捕获 11 个在模式层面本就显眼的账号，而误伤那 58%——新人、文档贡献者和顺手帮忙的路过者。我们由此得出结论：刷量是账号贡献历史的**模式属性**（集中度、模板化、对单一目标的重复），而非任何单个 diff 的**事件属性**；作用于单个事件的检测器测量的是错误的量。

关于拒绝率的结果与之平行。在拥有至少 10 个已裁决 PR 的 2,003 名开发者中，维护者拒绝率的中位数为 **2%**，第 90 百分位为 13%。因此通过率几乎不携带判别信号——在"有 PR 被合并"的条件下，此类贡献者提交的内容几乎全部会被合并——只有极端的拒绝率（规则的标记阈值为 50% 以上）具有信息量。

### 3.4 红旗构成：空缺主导，操纵罕见

用当前引擎对全部 3,444 个深度样本账号重新计算，**17%** 触发至少一条红旗。**图 3** 展示其构成。

![红旗流行度](/blog/we-scored-19000-github-accounts/red-flags.svg "图 3：深度样本中的红旗流行度（n = 3,444）。空缺类红旗比操纵类红旗高一个数量级。")

流行度最高的三条红旗——`ghost_profile`（幽灵档案，10%）、`no_original_work`（无原创工作，8.2%）、`mostly_forks`（几乎全为 fork，7.2%）——描述的是实质内容的缺失，而非操纵。主动欺骗类红旗低一个数量级：`templated_pr_flooding` 为 0.5%，`trivial_pr_farming` 为 0.3%，`follow_farming` 为 0.1%。star 注水（高 star 而 fork 与 issue 接近于零）在本样本中约等于零；不过自选择很可能压低了该数字——购买过 star 的账号不太可能主动提交自己接受评分。

对于构建在 GitHub 数据之上的信任工具，这一构成意味着两种性质不同的失败模式，需要不同的检测器与不同的容错标准：高频而低成本的情形（"这里什么都没有"），与罕见而高代价的情形（"这里有人伪造了内容"）。

### 3.5 账号年龄：不可伪造的信号

![账号年龄与中位数分数](/blog/we-scored-19000-github-accounts/age-vs-score.svg "图 4：按账号年龄划分的中位数最终分数。该关系在所有年龄桶上均为单调。")

中位数分数随账号年龄单调上升，从**不满一年账号的 18 分升至十年以上账号的 86 分**，中间无任何回落（**图 4**）。幸存者偏差对该关系有贡献——出现在样本中的老账号是仍在使用的老账号——但效应的方向本身即具有信息量。长周期一致性的每一个组成部分（多年的活跃、带有自然累积 star 的老仓库、跨越多个版本的贡献历史）恰恰是造假无法压缩的：star 与关注者可在数小时内获得，而一个 2015 年注册、活跃九年的账号无法在 2026 年被铸造。这一不对称性支持了规则中对账号成熟度与活跃年限的加权，也与我们观察到的"被标记的洪泛账号集中于年轻账号"一致。

作为附带观察，高分账号（最终分数 ≥ 60）的主语言构成符合大家熟知的生态趋势：**TypeScript（520）、Python（460）、JavaScript（395）** 领先，Rust（225）高于 Go（189）、C（184）与 Java（159）（**图 5**）。

![高分者的语言](/blog/we-scored-19000-github-accounts/languages.svg "图 5：60 分及以上账号的主语言分布。")

## 4. 讨论

上述结果指向贡献完整性工具的三条设计原则。

**检测模式，而非事件。** 3.2 节的双峰分离与 3.3 节 58% 对 0.4% 的差距表明，单个贡献几乎不携带造假信号，而账号级聚合可以干净地分离两类人群。基于单事件的启发式规则会把误报最大化地施加在开源项目最需要留住的贡献者——新人与低频帮助者——身上。

**区分空缺与操纵。** 未通过完整性检查的账号多数是空的，而非对抗性的（3.4 节）。混淆二者会夸大表观欺诈率，并错配审查资源。

**为时间赋权。** 纵向一致性是所考察信号中唯一无法通过花钱降低获取成本的信号（3.5 节），因而是任何可信度评分的自然锚点。

依据这些原则，我们正在构建一个帮助维护者分诊垃圾 PR 的 GitHub App，将 PR 级特征（diff 大小、模板相似度）与作者级历史相结合。鉴于 3.3 节记录的误报风险，该工具不会自动关闭任何 PR，而只呈现证据、由人做决定。我们欢迎受影响仓库的维护者提供其收到的垃圾 PR 样例。

## 5. 局限性

- **自选择样本。** 网站用户加活跃开源组织成员，偏向真实活跃。GitHub 全站的造假率很可能高于本文报告值；本文的百分比是已过滤人群内的下限。
- **规模。** 18,947 个已评分账号，3,444 个含深度指标。分布形状在多次重跑中稳定，但样本不能代表 GitHub 全体。
- **仅做聚合报告。** 所有模式均以聚合形式报告，不指认任何具体账号。红旗阈值全部公开，以便维护者结合上下文自行判断。
- **引擎版本。** 红旗流行度以当前评分器对每个账号最新原始快照重新计算；存储的垃圾分数反映扫描当时的引擎版本。二者均包含在[公开聚合数据](/blog/we-scored-19000-github-accounts/data.json)中。

## 6. 可复现性

全部评分逻辑均为确定性且以 AGPL 协议开源：[github.com/hikariming/ghfind](https://github.com/hikariming/ghfind)。同一引擎通过 `npm install ghfind` 与 `pip install ghfind` 分发，既可调用公开 API（[OpenAPI 规范](https://ghfind.com/openapi.json)），也可携带用户自己的 GitHub token 完全在本地运行。本文所有图表背后的聚合统计见 [data.json](/blog/we-scored-19000-github-accounts/data.json)。

*单个账号可在 [ghfind.com](https://ghfind.com) 进行评分。*
