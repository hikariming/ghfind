我们的数据集里有一个账号，合并 PR 的履历让大多数开发者羡慕——是真的被合并了，不只是提交；通过率看上去无懈可击。但它近期 PR 标题里有 97% 是几乎一模一样的模板变体，而且绝大多数都打向同一个它并不拥有的热门仓库。单看每一个 PR 都说得过去；但**模式**骗不了人：一台 AI 辅助的贡献刷量机器，正在速通一份"看起来可信"的 GitHub 履历。

这个账号就是我们开始做统计的原因。过去几周，[ghfind](https://ghfind.com) 用一套确定性引擎给 **18,947 个公开 GitHub 账号**打了分——评分核心不用大模型，每个数字都可以从公开数据复现。其中 3,444 个账号我们保留了完整的原始指标快照（PR 级样本、仓库质量特征、活跃形态），这让我们可以回答一个我们还没见过有人用数据完整回答的问题：

**GitHub 上的贡献造假到底有多普遍？它长什么样？**

短版本：比舆论想象的罕见得多；一旦出现，比想象的极端得多；而且只要你看的是"模式"而不是"单个事件"，把它和真实工作分开几乎不费力。

## 我们怎么评分（好让你自己判断这份数据）

引擎实现了一套确定性打分规则，六个维度合计 100 分，红旗信号扣分。没有模型调用、没有玄学；同样的输入永远得到同样的分数。核心代码以 AGPL 协议开源在 [ghfind 仓库](https://github.com/hikariming/ghfind)，网站、npm/PyPI SDK 和本文的分析走的是同一套代码路径。

| 维度 | 满分 | 奖励什么 |
|---|---|---|
| 贡献质量 | 27 | 合并 PR 数（对数刻度）、通过率、issue 参与 |
| 生态影响力 | 20 | 对高星仓库的实质性 PR、维护者深度 |
| 原创项目质量 | 18 | star 数 × 仓库实质（高星空壳得分很低） |
| 活跃真实性 | 17 | 持续的近期活跃、活动类型多样性 |
| 账号成熟度 | 10 | 账号年龄、真实活跃年数 |
| 社区影响力 | 8 | 关注者（对数刻度）、关注/被关注比合理性 |

在此之上还有十二种确定性红旗扣分——比如 `templated_pr_flooding`（模板化 PR 洪泛）、`trivial_pr_farming`（垃圾 PR 刷量）、`follow_farming`（互关刷粉）、`possible_star_inflation`（疑似 star 注水）。完整阈值都在仓库里；本文除了后面会讲到的一个内部反刷分数外，不使用任何隐藏数据。

看图之前先坦白两个前提。第一，这不是 GitHub 的随机抽样：样本是在烧烤站上主动给自己评分的人，加上我们从活跃开源组织收录的开发者。样本天然**偏向真实、活跃的开发者**——所以下文所有造假比例，都应该读作"一个已被过滤过的人群里的下限"。第二，1.9 万个账号是不少扫描量，但只是 GitHub 的一小片切面；我们发布的是形状，不是普查。

## 一半账号的分数低于 40

![19k 账号的最终分数分布](/blog/we-scored-19000-github-accounts/score-distribution.svg "最终分数分布，5 分一桶。橙色是 70 分以上的段位。")

被评分账号的中位数刚过 40 分。**48.6%** 的账号低于 40 分——我们直白地把这一档标注为"低价值/疑似刷量"——而只有 **3.7%** 超过 90 分。即使在一个偏向真实活跃开发者的样本里，大多数公开 GitHub 档案也很单薄：最大的单个分数桶是 0–5 分，里面是没有原创作品、没有合并 PR、没有持续活跃的账号。

给你一个校准基准：深度样本里的中位数账号有 **27 个关注者、34 个 star、20 个合并 PR**——注册七年之后。如果你的数字长这样，你没有落后，你就是中位数。声望的重活都是长尾干的：p90 是 1,275 关注者、约 5,900 star；p99 是 19,000 关注者、约 100,000 star。

## 硬核刷量：占比 0.6%，但根本藏不住

在公开分数之外，引擎还会计算一个从未在任何地方展示过的内部 0–10 反刷分（spam/bot 可能性）——它存在的唯一目的是让榜单安静地抵抗操纵。这是它的分布第一次公开：

![隐藏反刷分分布](/blog/we-scored-19000-github-accounts/spam-score.svg "77% 的账号是干净的 0 分。尾部很小——但很扎眼。")

- **77%** 的账号得分正好是 0——完全没有刷量信号。
- **0.58%**（18,934 个中的 110 个）达到 3 分以上，我们的"明确在刷"红线。
- **12 个账号**达到 7 分以上，铁证如山区。

刷子容易被抓，靠的不是量大，而是**集中**。在 2,122 个近期有至少 10 个 PR 的开发者里，模板化标题占比的**中位数是 7%**。人类都会小小地重复自己（"fix typo"、"bump deps"）。而被标记的洪泛账号全部超过 **50%**，最高到 **97%**（整个人群的 99 分位也才 72%）——"正常重复"和"批量生成"之间几乎不存在中间地带。这个分布不是一条斜坡，是两座孤岛。抓这些账号不需要机器学习；需要的是一次多看几个 PR。

## 改变我们路线图的发现：琐碎 PR 是常态

这是最让我们意外的数字。在 2,558 个有近期合并 PR 样本的开发者中，**58% 至少有一个琐碎 PR**——被合并进别人 200+ star 仓库的 ≤5 行改动。在名仓库里修一个 typo，看来是某种成人礼。

但近期合并 PR **以琐碎为主**、且数量达到 10 个以上的开发者呢？**0.4%**（11 个账号）。

这个落差就是反垃圾工具的全部设计任务书。任何惩罚"小 PR"的启发式规则，都会为了抓住那 11 个从来就不隐蔽的刷子，而误伤那 58%——新人、文档修复者、开源赖以生存的路过帮忙的人。刷量是一种**模式属性**（集中度、模板化、对单一目标的重复），不是**事件属性**（一个小 diff）。任何看到单个 PR 就开火的垃圾检测器，量错了东西。

同样的逻辑也洗清了另一个流行怀疑：PR 被拒。在 2,003 个有 10 个以上已裁决 PR 的开发者里，**维护者拒绝率的中位数是 2%**，p90 也只有 13%。对于 PR 能被合并的人，维护者几乎什么都合——所以"高通过率"是个几乎没用的证书，只有**极端**的拒绝率（我们的阈值是 50% 以上）才携带信号。

## 大多数"看着像假"的账号不是在刷——是空的

用当前引擎重跑全部 3,444 个深度样本账号，**17%** 触发至少一面红旗。有意思的是构成：

![红旗流行度](/blog/we-scored-19000-github-accounts/red-flags.svg "「空」比「假」高出一个数量级。")

前三名红旗——`ghost_profile`（幽灵档案，10%）、`no_original_work`（无原创作品，8.2%）、`mostly_forks`（几乎全是 fork，7.2%）——描述的是**缺席**，不是操纵。主动造假（模板化 PR 洪泛 0.5%、垃圾 PR 刷量 0.3%、互关刷粉 0.1%）要低一个数量级。star 注水——高 star 但 fork 和 issue 几乎为零——在这个样本里约等于零，不过我们估计是自选择把它藏起来了：买了 star 的人不会来烧烤自己。

如果你在 GitHub 数据之上构建信任工具，这个构成很重要：你常遇到的失败模式是"这里什么都没有"，而昂贵但罕见的失败模式是"有人在这里伪造了点什么"。两者需要不同的检测器，和差别极大的容错率。

## 唯一花钱买不到的信号：十年

![账号年龄与中位数分数](/blog/we-scored-19000-github-accounts/age-vs-score.svg "中位数分数随账号年龄单调爬升。")

中位数分数从**不满一年的账号的 18 分，爬到十年以上账号的 86 分**——单调递增，没有一次回落。其中有幸存者偏差（还在被评分的老账号，是还在被使用的老账号）。但这恰恰也是重点：长周期一致性的每一个组成部分——多年的活跃、带着自然 star 的老仓库、跨越多个版本的贡献图——正是刷量机器无法压缩的东西。star 今天下午就能买到，粉丝今晚就能刷够。但一个 2015 年注册、活跃了九年的账号，无法在 2026 年被铸造出来。

这就是为什么账号成熟度和活跃年份在规则里占有实打实的权重，也是为什么洪泛刷子几乎都聚集在年轻账号里——它们在试图伪造唯一无法快速伪造的东西。

顺带一提，高分选手的语言构成不会让任何关注过去五年的人意外——在 60 分以上的账号里，**TypeScript（520）、Python（460）、JavaScript（395）** 领跑，Rust（225）已经稳稳压过 Go（189）、C（184）和 Java（159）：

![高分者的语言](/blog/we-scored-19000-github-accounts/languages.svg "60 分以上账号的主力语言。")

## 我们要拿这些数据做什么

上面的一切都是确定性、可复现的：`npm install ghfind` 或 `pip install ghfind` 就能拿到同一套评分引擎——既可以打我们的 API，也可以带上你自己的 GitHub token 完全在本地跑（评分核心直接打包在包里），[代码以 AGPL 开源](https://github.com/hikariming/ghfind)。还有公开评分 API 和 [OpenAPI 规范](https://ghfind.com/openapi.json)，想把分数挂在个人主页上也有现成的 README 徽章。

基于这个引擎，我们下一步要做的正是这份数据主张的事：一个帮开源维护者分诊垃圾 PR 的 GitHub App——把 PR 级信号（diff 大小、模板相似度）和作者级历史结合起来，并且**绝不自动关闭任何 PR**，因为上面那个 58% 的发现说得很清楚：误伤打击的恰恰是开源最需要的人。如果你维护的仓库正在被刷，我们很想知道你那边的垃圾长什么样。

## 局限性，坦白讲

- **自选择样本。** 烧烤站用户加上活跃开源组织成员，偏向真实活跃。全 GitHub 的刷量率肯定比我们的数字高；请把我们的百分比当作"已过滤人群中的下限"。
- **规模。** 1.9 万个已评分账号，3,444 个有深度指标。形状在多次重跑中稳定，但这是一台望远镜，不是一次普查。
- **不点名。** 所有模式都以聚合形式报告。重点是数据的形状，不是任何具体账号——红旗阈值的意义正是让维护者带着上下文自己做判断。
- **引擎版本。** 红旗流行度是用当前评分器对每个账号最新原始快照重算的；存储的反刷分反映扫描当时的引擎版本。两者都在[原始聚合数据](/blog/we-scored-19000-github-accounts/data.json)里，欢迎验算。
