**要旨。** GitHub 上の捏造された活動——テンプレート化されたプルリクエスト、購入されたスター、相互フォローネットワーク——への懸念は広く共有されているが、その発生率と構造に関する定量的証拠は乏しい。本研究では、決定論的かつオープンソースの採点基準（モデル呼び出しを行わず、同一の入力からは同一のスコアが得られる）を用いて 18,947 の公開 GitHub アカウントをスコアリングし、うち 3,444 アカウントについては完全な生指標スナップショットを保持する詳細サンプルとした。得られた知見は次の通りである。(i) 本サンプルにおいて捏造は稀であり、保守的に設定したファーミング閾値を超えるアカウントは 0.58% にすぎない。(ii) 捏造は発生する場合には極端であり、通常の行動から双峰的に分離している——フラグが立ったアカウントのテンプレート化 PR タイトル比率はいずれも 50% 超（最大 97%）であるのに対し、母集団の中央値は 7% である。(iii) 低労力の貢献は疑わしいものではなく正常である——直近のマージ済み PR サンプルを持つ開発者の 58% は軽微な外部 PR を少なくとも 1 件有するが、大量かつ過半が軽微というパターンを示すのは 0.4% にとどまる。(iv) 「疑わしく見える」プロファイルの支配的な形態は、操作されたものではなく空である——欠如型のレッドフラグは操作型のフラグを一桁上回る。スコアの中央値はアカウント年齢とともに単調に増加し、1 年未満の 18 点から 10 年超の 86 点に達する。これは、長期にわたる一貫性が最も捏造しにくい信号であることと整合的である。これらの結果から、スパム検出ツールの設計への含意を論じる。特に、ファーミングとは個々の貢献というイベントレベルの性質ではなく、アカウント履歴というパターンレベルの性質である。

## 1. はじめに

開発者の信頼性評価は、公開された GitHub 活動への依存を強めている。採用パイプラインは候補者のプロフィールを審査し、オープンソースのメンテナは見知らぬ貢献者からのプルリクエストをトリアージし、下流のツールは見かけ上のインパクトによってアカウントをランク付けする。これらの用途はいずれも、その基礎となる信号を捏造する誘因を生み出す。スター売買、テンプレート化されたプルリクエスト・キャンペーン、相互フォロー・スキームに関する逸話的報告は数多いが、そうした捏造がどの程度の頻度で発生し、統計的にどのような形態をとるのかについての体系的な測定は存在しない。

本データセットに含まれる一例がこの現象をよく示している。あるアカウントのマージ済み PR の記録は、通常であれば有力な貢献者であることを示すものである。すなわち、（単に作成されただけでなく）マージされた多数のプルリクエストと、ほぼ完璧な受理率である。しかし仔細に検討すると、直近の PR タイトルの 97% が互いにほぼ同一のテンプレート変種であり、その過半が当該アカウントの所有しない単一の人気リポジトリに向けられていることが分かる。個々のプルリクエストはどれも異常ではなく、異常は集計パターンのレベルにのみ存在する。捏造はイベント単位では不可視でありながら集計すれば顕著でありうる——この観察が本研究の動機である。

本研究では次の三つの問いを立てる。

1. **発生率。** 公開 GitHub アカウントにおいて、捏造された貢献活動はどの程度一般的か。
2. **構造。** 捏造が発生する場合、それは通常の貢献行動と統計的にどう異なるか。
3. **構成。** 完全性ヒューリスティクスに引っかかるアカウントのうち、能動的な操作を反映するものと、単なる不活発や独自の成果の欠如を反映するものの割合はどうか。

これらに答えるため、決定論的な採点基準（[ghfind](https://ghfind.com)。そのスコアリングコアは AGPL の下でオープンソースである（[リポジトリ](https://github.com/hikariming/ghfind)））を用いて 18,947 の公開アカウントをスコアリングし、PR 単位のサンプル、リポジトリ品質特徴、活動形状統計を含む完全な生指標スナップショットを保持する 3,444 アカウントの詳細サンプルを分析する。各図の基礎となる集計データはすべて本稿とともに公開する（[data.json](/blog/we-scored-19000-github-accounts/data.json)）。

要約すれば、本サンプルにおける捏造は世間の言説が示唆するよりも大幅に稀であり、存在する場合には巧妙ではなく極端であり、パターンレベルの単純な閾値のみで通常の活動から分離可能である。

## 2. データと手法

### 2.1 採点基準

本エンジンは、合計 100 点となる六つの次元にわたる決定論的な採点基準を実装し、レッドフラグ信号に対して加算式のペナルティを課す。モデル呼び出しは一切行わず、スコアは公開 GitHub データから完全に再現可能である。ghfind のウェブサイト、npm/PyPI の SDK、および本分析で用いられるスコアは、同一のコードパスによって算出されている。

| 次元 | 満点 | 評価される信号 |
|---|---|---|
| 貢献品質 | 27 | マージ済み PR 数（対数スケール）、受理率、issue への参加 |
| エコシステムへの影響 | 20 | 高スターリポジトリへの実質的な PR、メンテナとしての関与の深さ |
| 独自プロジェクトの品質 | 18 | リポジトリの実質性で重み付けしたスター数 |
| 活動の真正性 | 17 | 持続的な直近の活動、活動種別の多様性 |
| アカウント成熟度 | 10 | アカウント年齢、実際に活動した年数 |
| コミュニティ影響力 | 8 | フォロワー数（対数スケール）、フォロワー/フォロー比の妥当性 |

12 の決定論的なレッドフラグ規則が減点を行い、そこには `templated_pr_flooding`、`trivial_pr_farming`、`follow_farming`、`possible_star_inflation` が含まれる。正確な閾値はリポジトリで参照できる。公開スコアに加えて、エンジンはリーダーボードの完全性を保護するために用いられる 0–10 スケールの内部スパム/ボット尤度スコアを算出しており、第 3.2 節でその分布を初めて報告する。それ以外の非公開データは本分析に一切用いていない。

**定義。** 変更行数が 5 行以下で、かつ作者が所有しない 200 スター以上のリポジトリにマージされたプルリクエストを*軽微な PR* と呼ぶ。アカウントの*テンプレート化タイトル比率*とは、直近の PR タイトルのうち、互いにほぼ同一のテンプレート変種である割合を指す。

### 2.2 サンプル構成と既知のバイアス

サンプルは、(a) ghfind のウェブサイトを通じて自発的に自分のアカウントをスコアリングしたユーザーと、(b) 活発なオープンソース組織から取り込んだ開発者から構成される。この設計の二つの性質が解釈を制約する。第一に、サンプルは自己選択的であり、真正で活発な開発者に偏っている。したがって以下で報告するすべての捏造率は、GitHub 全体の推定値ではなく、**既にフィルタリングされた母集団内での下限**として読まれるべきである。第二に、スコアリング済み 18,947 アカウント（うち詳細指標を持つのは 3,444）という規模は分布の形状を特徴付けるには十分だが、GitHub 全体から見れば無視できる割合であり、本稿が報告するのは形状であって全数調査ではない。

## 3. 結果

### 3.1 スコア分布

![19k アカウントにわたる最終スコア分布](/blog/we-scored-19000-github-accounts/score-distribution.svg "図 1：最終スコアの分布（5 点刻み、n = 18,947）。オレンジの帯は 70 点以上の階層を示す。")

*図 1* は最終スコアの分布を示す。中央値は 40 点をわずかに上回る。**48.6%** のアカウントが 40 点未満（採点基準が低価値または水増しの疑いと分類する階層）であり、90 点を超えるのは **3.7%** にすぎない。最も人数の多い区間は 0–5 点であり、独自の成果もマージ済みプルリクエストも持続的な活動も持たないアカウントで構成される。活発な開発者に偏ったサンプルにおいてすら、大半の公開プロフィールは中身が薄い。

較正のために述べると、詳細サンプルにおける中央値のアカウントは、アカウント年齢の中央値 7 年で、**フォロワー 27、スター合計 34、マージ済み PR 20 件**である。評判に関わる指標は上位テールに強く集中しており、第 90 パーセンタイルはフォロワー 1,275、スター約 5,900、第 99 パーセンタイルはフォロワー 19,000、スター約 100,000 である。

### 3.2 ファーミングの発生率と構造

![隠しスパムスコアの分布](/blog/we-scored-19000-github-accounts/spam-score.svg "図 2：内部の 0–10 スパム尤度スコアの分布（n = 18,934）。77% のアカウントはちょうど 0 点である。")

*図 2* は、算出対象となった 18,934 アカウントにわたる内部スパム尤度スコアの分布を報告する。

- **77%** のアカウントはちょうど 0 点であり、いかなる種類のファーミング信号も持たない。
- **0.58%**（110 アカウント）が 3 点以上であり、これを明確なファーミングとみなす閾値とする。
- **12 アカウント**が 7 点以上である。

ファーミング・アカウントの分離可能性は、その量ではなく*集中度*に由来する。直近のプルリクエストを 10 件以上持つ 2,122 名の開発者のうち、テンプレート化タイトル比率の中央値は **7%** である——通常の開発者もほどほどに同じことを繰り返す（「fix typo」「bump deps」）。一方、テンプレート大量投稿のフラグが立ったアカウントはすべて比率が **50%** を超えて **97%** にまで達しているのに対し、全母集団の第 99 パーセンタイルは 72% である。この分布は事実上双峰的であり、通常の反復と一括生成の間に連続体は存在しない。これらのアカウントの検出に学習済みモデルは不要であり、必要なのはアカウントごとに複数のプルリクエストにまたがる集計だけである。

### 3.3 軽微なプルリクエストは正常であり、軽微 PR ファーミングは正常ではない

直近のマージ済み PR サンプルを持つ 2,558 名の開発者のうち、**58%** は第 2.1 節で定義した軽微な外部 PR を少なくとも 1 件有する。著名なリポジトリへの小さなドキュメント修正や誤字修正は、明らかにオープンソース参加の日常的な一部であり、それは本来強力な貢献者にとっても同様である。

対照的に、直近のマージ済み PR の*過半*が軽微であり、かつそのような PR を 10 件以上持つ開発者は、サンプルの **0.4%**（11 アカウント）にとどまる。

この二つの数字の隔たりは、不正対策ツールの設計に直接的な帰結をもたらす。小さなプルリクエストそれ自体を罰するヒューリスティクスはいずれも、パターンレベルで既に顕著であった 11 のアカウントを捕捉するために、58% の側——新規参入者、ドキュメント貢献者、偶発的な協力者——を罰することになる。我々は、ファーミングとはアカウントの貢献履歴の**パターン特性**（集中度、テンプレート化、単一ターゲットへの反復）であって、個々の diff の**イベント特性**ではなく、単一イベントに対して動作する検出器は誤った量を測定している、と結論する。

拒否率についても並行する結果が成り立つ。結果の確定した PR を 10 件以上持つ 2,003 名の開発者のうち、メンテナによる拒否率の中央値は **2%**、第 90 パーセンタイルは 13% である。したがって受理率の判別信号はわずかであり——そもそもマージ済み PR を持つことを条件とすれば、そうした貢献者の提出物はほぼすべてマージされる——極端な拒否率（採点基準は 50% 超でフラグを立てる）のみが情報を持つ。

### 3.4 レッドフラグの構成：欠如が操作を圧倒する

現行エンジンを詳細サンプル全 3,444 アカウントに対して再実行すると、**17%** が少なくとも一つのレッドフラグに該当する。*図 3* はその構成を示す。

![レッドフラグの出現率](/blog/we-scored-19000-github-accounts/red-flags.svg "図 3：詳細サンプルにおけるレッドフラグの出現率（n = 3,444）。欠如型フラグは操作型フラグを一桁上回る。")

出現率の最も高い三つのフラグ——`ghost_profile`（10%）、`no_original_work`（8.2%）、`mostly_forks`（7.2%）——が記述するのは操作ではなく実質の欠如である。能動的欺瞞のフラグは一桁少なく、`templated_pr_flooding` が 0.5%、`trivial_pr_farming` が 0.3%、`follow_farming` が 0.1% である。スター水増し——フォークと issue がほぼゼロであるにもかかわらずスター数が多い——は本サンプルでは四捨五入するとゼロになるが、自己選択がこれを抑制している可能性は十分にある。購入したスターを持つアカウントが自らスコアリングに応じることは考えにくい。

GitHub データの上に構築される信頼ツールにとって、この構成は、異なる検出器と異なる誤り許容度を要する質的に異なる二つの失敗モードを含意する。すなわち、高頻度で低コストのケース（「ここには何もない」）と、稀で高コストのケース（「ここにあるものは製造された」）である。

### 3.5 偽造不可能な信号としてのアカウント年齢

![アカウント年齢別のスコア中央値](/blog/we-scored-19000-github-accounts/age-vs-score.svg "図 4：アカウント年齢別の最終スコア中央値。この関係はすべての年齢区間にわたって単調である。")

スコアの中央値はアカウント年齢とともに単調に上昇し、**1 年未満のアカウントの 18 点から 10 年を超えたアカウントの 86 点**まで、途中での低下は見られない（*図 4*）。生存者バイアスはこの関係に寄与している——サンプルに現れる古いアカウントは、今も使われている古いアカウントである——が、効果の方向それ自体が情報を持つ。長期にわたる一貫性のあらゆる構成要素（活動年数、有機的にスターを蓄積してきた古いリポジトリ、多数のリリースにまたがる貢献履歴）は、まさに捏造が圧縮できないものである。スターとフォロワーは数時間で獲得できるが、9 年間の活動を持つ 2015 年のアカウントを 2026 年に新造することはできない。この非対称性は、採点基準がアカウント成熟度と活動期間に重みを置くことを正当化するものであり、フラグの立った大量投稿アカウントが若いアカウントに集中するという我々の観察とも整合的である。

副次的な観察として、高スコアアカウント（最終スコア ≥ 60）の主要言語構成は馴染み深いエコシステムの趨勢に従っており、**TypeScript（520）、Python（460）、JavaScript（395）** が先頭に立ち、Rust（225）が Go（189）、C（184）、Java（159）を上回る（*図 5*）。

![高スコア者の言語](/blog/we-scored-19000-github-accounts/languages.svg "図 5：スコア 60 以上のアカウントにおける主要言語。")

## 4. 考察

以上の結果から、貢献の完全性を扱うツールの三つの設計原則が導かれる。

**イベントではなくパターンを検出せよ。** 第 3.2 節の双峰的分離と第 3.3 節における 58% 対 0.4% の隔たりは、個々の貢献がほとんど捏造信号を持たない一方、アカウントレベルの集計はきれいに分離することを示している。単一イベントのヒューリスティクスは、オープンソースプロジェクトが最も引き留める必要のある貢献者——新規参入者と低頻度の協力者——に対してこそ偽陽性を最大化する。

**欠如と操作を区別せよ。** 完全性チェックに落ちるアカウントの大半は敵対的なのではなく空である（第 3.4 節）。両者の混同は見かけの不正率を膨張させ、レビューの労力配分を誤らせる。

**時間に重みを置け。** 縦断的な一貫性は、検討した信号の中で唯一、支出によって獲得コストを下げられない信号であり（第 3.5 節）、あらゆる信頼性スコアの自然な錨となる。

これらの原則に導かれ、我々はメンテナによるスパム・プルリクエストのトリアージを支援する GitHub App を構築している。これは PR レベルの特徴（diff サイズ、テンプレート類似度）と作者レベルの履歴を組み合わせるものである。第 3.3 節で記録した偽陽性リスクに鑑み、本ツールはプルリクエストを自動クローズせず、人間の判断のために証拠を提示する。影響を受けているリポジトリのメンテナ各位には、受け取っているスパムの実例の提供を呼びかけたい。

## 5. 限界

- **自己選択サンプル。** ウェブサイトのユーザーと活発なオープンソース組織のメンバーは、真正かつ活発な側に偏る。GitHub 全体の捏造率は本稿の報告値より高い可能性が十分にあり、本稿のパーセンテージはフィルタリングされた母集団内での下限である。
- **規模。** スコアリング済み 18,947 アカウント、うち詳細指標を持つのは 3,444。分布の形状は再実行を通じて安定しているが、サンプルは GitHub 全体を代表するものではない。
- **集計報告のみ。** すべてのパターンは集計形式で報告し、個別のアカウントは特定しない。フラグの閾値は公開されており、メンテナは文脈を踏まえてそれらを適用できる。
- **エンジンのバージョン管理。** レッドフラグの出現率は、各アカウントの最新の生スナップショットに対して現行スコアラーで再計算したものである。保存されているスパムスコアはスキャン時点のエンジンバージョンを反映する。いずれも[公開集計データ](/blog/we-scored-19000-github-accounts/data.json)に含まれる。

## 6. 再現可能性

すべてのスコアリングロジックは決定論的であり、[github.com/hikariming/ghfind](https://github.com/hikariming/ghfind) においてオープンソース（AGPL）である。同一のエンジンは `npm install ghfind` および `pip install ghfind` を通じて配布されており、公開 API（[OpenAPI 仕様](https://ghfind.com/openapi.json)）に対しても、ユーザー自身の GitHub トークンを用いて完全にローカルでも実行できる。本稿のすべての図の背後にある集計統計は [data.json](/blog/we-scored-19000-github-accounts/data.json) として入手可能である。

*個別のアカウントは [ghfind.com](https://ghfind.com) でスコアリングできる。*
